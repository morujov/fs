**Тип продукта:** доска объявлений (tablón de anuncios / clasificados), НЕ маркетплейс
**Стек:** Laravel 13 + **MySQL 5.7** + Blade + Alpine.js + Tailwind, Filament v3 для админки

## 0. Позиционирование — прочитать первым

Сайт **не является стороной сделки**. Мы публикуем объявления и показываем контакты. Деньги, передача номера, споры — вне платформы. Это должно быть зафиксировано в трёх местах: Aviso legal, Términos de uso, и футер каждой страницы.

**Важно понимать про предмет торговли.** Юридически в Испании номер не является собственностью абонента — он принадлежит нумерационному плану (CNMC) и закреплён за оператором. «Продажа номера» на практике = **portabilidad**: покупатель переносит номер на себя с согласия текущего владельца. Мы этого не делаем и не гарантируем. В шапке формы подачи и на карточке объявления нужен дисклеймер: *«Anuncio de cesión de línea. La transferencia se realiza mediante portabilidad entre las partes y sus operadores. Este sitio no interviene en la operación.»*

Практическое следствие для архитектуры: у объявления обязателен **оператор** и признак **libre / con permanencia** — без них объявление бесполезно покупателю.

## 4. Wildcard-поиск — ядро продукта

### Правила ввода

- Номер: ровно 9 цифр, первая — `6` или `7` (испанский мобильный).
- Маска поиска: цифры + `?` (одна любая цифра). Пустая позиция = `?`.
- Разрешённый ввод: `^[0-9?]{0,9}$`. Всё остальное — отклонить **до** SQL.

### Преобразование

```php
// '6??12??34' → LIKE '6__12__34'
$pattern = preg_replace('/[^0-9?]/', '', $input);      // 1. жёсткая санитизация
$pattern = str_pad($pattern, 9, '?');                   // 2. добить справа
$like    = strtr($pattern, ['?' => '_']);               // 3. '?' → SQL '_'
Listing::where('status','active')->where('msisdn','LIKE',$like);
```

**Безопасность:** `%` и `_` от пользователя обязаны быть отфильтрованы шагом 1. Если пользователь введёт `%`, он получит выгрузку всей базы одним запросом. Это дыра, а не косметика.

### UX

Девять отдельных боксов-ячеек (как код 2FA), навигация стрелками, `?` вводится клавишей `?` или пробелом. Плюс быстрые пресеты: *«termina en 777»*, *«repetidos»*, *«capicúa»*, *«escalera»* — они же становятся SEO-посадочными страницами.

### Производительность и путь отхода

`LIKE '6__12__34'` использует индекс по префиксу (первый символ известен). `LIKE '___12____'` — не использует, будет full scan. На CHAR(9) и объёме до ~200 000 строк это ~30–60 мс, приемлемо. Когда перестанет хватать:

1. **План B:** 9 generated-колонок `d1..d9 TINYINT` + составные индексы на частые комбинации.
2. **План C:** держать все активные `msisdn` в памяти (200k × 9 байт ≈ 2 МБ) и фильтровать в PHP/APCu, БД — только за деталями.

Не оптимизировать заранее. Но `NumberPatternQuery` изолировать в отдельный класс, чтобы замена движка не трогала контроллеры.

### Прочие фильтры

Провинция (multi), оператор (multi), цена от/до, `condition`, `libre/permanencia`, `prepago/contrato`, тип продавца, только с фото логотипа. Сортировка: новые / цена ↑ / цена ↓. **Пагинация 20/стр.**

## 4A. Авторизация и раскрытие контактов

**Решение:** контакты видны наполовину всем; полностью — только после входа через Google Sign-In. Авторизация в проекте одна — Google OAuth 2.0. Паролей нет вообще.

### Почему только Google, без паролей

Убирает целый класс работы и рисков: регистрацию, подтверждение email, восстановление пароля, хранение хэшей, credential stuffing, утечку паролей. Email от Google приходит уже верифицированным — правило модерации «email не подтверждён» становится не нужным. Пакет: `laravel/socialite`, scopes только `openid email profile` (несенситивные → верификация OAuth-экрана проходит быстро, без аудита Google).

**Цена решения, которую надо принять осознанно:** часть покупателей отвалится на кнопке входа. Это осознанный обмен — вы меняете объём анонимных просмотров на защиту базы контактов и на email-базу для алертов. При нулевой монетизации email-база — ваш главный актив, так что обмен выгодный.

**Что НЕ гейтить:** просмотр списка, wildcard-поиск, все фильтры, карточка объявления, цена, провинция, оператор. Всё это должно быть доступно анониму и Googlebot — иначе SEO мёртв, а SEO это весь ваш трафик. Гейт стоит **только на кнопке раскрытия контакта**.

### Правила маскировки

Маска считается **на сервере, в PHP**. В HTML уходит только замаскированная строка.

| Поле | Аноним | После входа |
|------|--------|-------------|
| Телефон продавца | `6** ** ** **` | `612 34 56 78` |
| Имя | `Juan M.` | `Juan Martínez` |
| Email | `j••••@gmail.com` | `juan@gmail.com` |
| WhatsApp | кнопка неактивна | `wa.me` deep-link |

**Почему маскируем телефон продавца по максимуму, до первой цифры.** Продаваемый номер (`listings.msisdn`) виден всем целиком и никогда не маскируется — это товар, витрина и весь SEO. Телефон продавца — только канал связи, для оценки товара он не нужен. Значит полное сокрытие не стоит нам ни конверсии, ни SEO, а скрейперу не даёт ничего. Показывать его хвост было бы бессмысленным подарком.

**Главная ошибка, которую нельзя допустить:** отрендерить полный контакт в HTML и спрятать CSS-ом (`blur`, `opacity`, `::before`) или JS-ом. Так делают 90% досок объявлений, и это вскрывается через `Ctrl+U` за пять секунд — то есть защиты нет вообще. Полное значение приходит **только** в ответе AJAX-эндпоинта, **только** после проверки сессии на сервере. То же касается `wa.me`-ссылки и `mailto:` — они не существуют в разметке до раскрытия.

### Эндпоинт раскрытия

```
POST /api/listings/{listing}/contact
  middleware: auth (Google-сессия), throttle:contact-reveal
  → 401 если гость          — фронт показывает модалку «Entra con Google»
  → 429 при превышении лимита
  → 200 {name, phone, email, whatsapp_url}
  побочно: запись в contact_reveals + инкремент listings.contact_reveals
```

### Лимиты и антибот

Google-аккаунт стоит ноль евро — сам по себе OAuth скрейпинг **не останавливает**, он только повышает цену. Останавливают лимиты поверх него:

| Правило | Порог | Действие |
|---------|-------|----------|
| Раскрытий в минуту / аккаунт | 5 | 429 |
| Раскрытий в сутки / аккаунт | 20 | 429 + Turnstile |
| Раскрытий в сутки / IP | 40 | 429, флаг в админку |
| Аккаунтов с одного IP за сутки | 3 | флаг в админку |
| Интервал между раскрытиями < 2 сек | 3 раза подряд | сигнатура бота → флаг |
| Раскрытий в сутки / аккаунт | 50 | автоблок аккаунта + очередь модератора |

Все пороги — в `settings`, правятся из админки без деплоя. Виджет на дашборде: «топ аккаунтов по раскрытиям за 24 ч» — так вы увидите скрейпер в первый же день.

### Продавцы

Заходят тем же Google Sign-In. `seller_type` (`private`/`shop`) выбирается при первой подаче объявления, не при входе. OTP-SMS на продаваемый номер остаётся — Google подтверждает личность, но не владение номером. Это разные проверки, одна другую не заменяет.

### GDPR-нюанс

Скрипт Google загружать **только на клик по кнопке входа**, не на каждой странице — иначе это сторонний трекер до получения согласия, и cookie-баннер не спасёт. Передачу данных в Google описать в Política de privacidad (получатель, цель, трансфер за пределы ЕС). Google при верификации OAuth-приложения сам потребует публичный URL политики конфиденциальности — она понадобится до первого входа, а не «потом».

## 5. Системная модерация — конвейер

`ModerationPipeline` — цепочка правил, каждое возвращает `pass | flag | reject` и пишет в `moderation_logs`. Итог: reject → отказ с причиной; flag → ручная очередь; всё pass → `active`.

**Правила (порядок важен):**

1. **Формат** — 9 цифр, первая `6` или `7`. Reject.
2. **Не служебный диапазон** — 112, 060, 900/901/902, 70X (numeración personal). Reject.
3. **Блок-лист** — таблица `blocklist_numbers` с wildcard-паттернами. Reject.
4. **Дубликат** — активное объявление с тем же `msisdn` существует. Reject.
5. **OTP не пройден** — `phone_verified_at IS NULL`. Держим в `pending`, не публикуем.
6. ~~Email не подтверждён~~ — **правило удалено.** Google отдаёт верифицированный email, проверять нечего.
7. **Цена** — в диапазоне 1–50 000 € или `is_negotiable`. Reject вне диапазона.
8. **Контакты** — телефон валиден (E.164/ES), email валиден и не одноразовый (disposable-домены).
9. **Текст: контакты в описании** — регексп на телефон/email/ссылки/@ник внутри `description`. Это обход раскрытия контактов и главный вектор спама. Flag + автовырезание.
10. **Текст: запрещённое** — стоп-слова, ссылки, HTML. Flag.
11. **Rate limit** — >3 объявлений/сутки с аккаунта или >5 с IP. Flag.
12. **Honeypot + Turnstile** — скрытое поле заполнено или токен не прошёл. Reject молча.
13. **Возраст аккаунта** — младше 1 часа → flag на первое объявление.
14. **Магазин** — `shops.status != verified` → flag, объявление ждёт проверки магазина.
15. **Скоринг** — сумма флагов; `score >= порог` → ручная очередь, иначе публикуем.

**Редактирование опубликованного объявления → повторный прогон конвейера.** Иначе схема очевидна: опубликовать чистое, потом вписать спам.

### Требует проверки

- **Версия MySQL-сервера** (проверен только клиент). `SELECT VERSION();`
- **Исходящий интернет с хоста** — нужен для `composer install`. `curl -I https://repo.packagist.org`
- **`memory_limit` для CLI** — `composer install` прожорлив, нужно ≥512M
- **Расширения PHP:** `mbstring openssl pdo_mysql intl bcmath fileinfo ctype json curl`

### MySQL 5.7 — что это меняет

Версия 2015 года, снята с поддержки в октябре 2023. Laravel 13 с ней работает, наша схема тоже: `JSON`-колонки (5.7.8+) и generated-колонки поддерживаются. Чего нет: оконных функций, `CHECK`-констрейнтов, `NOWAIT`/`SKIP LOCKED`. В нашей схеме они не используются, так что переписывать нечего.

**Но:** проверьте в cPanel, нет ли опции MySQL 8 или MariaDB 10.6+. Если есть — переключаемся, это бесплатный апгрейд безопасности. Если нет — живём на 5.7, но помним, что это EOL-софт на периметре.

Заметка на будущее: `utf8mb4` + длинные `VARCHAR(255)` в индексах на 5.7 упираются в лимит ключа. В 5.7 `innodb_large_prefix` включён по умолчанию, так что проблемы быть не должно; если вылезет `Specified key was too long` — `Schema::defaultStringLength(191)` в `AppServiceProvider`.

### Node.js на хосте нет — и не понадобится

Собирать Tailwind/Vite на проде не нужно. Ассеты собираются **в моём сандбоксе** (Node 22 там есть) и коммитятся готовыми в `public/build`. На хост едет уже собранное. Проблема закрыта, ставить nvm не будем.

### Схема деплоя

```
~/Documents/numeros-es  (пишу я)
        ↓ git push
   GitHub (приватный репозиторий)
        ↓ git pull   ← в cPanel Terminal
/home4/clsthmmy/numeros-es        ← код, вне веб-корня
/home4/clsthmmy/public_html/test  ← только содержимое public/
        ↓
composer install --no-dev -o && php artisan migrate --force
```

### Чек-лист деплоя

1. **Document root.** Код в `~/numeros-es` (вне веб-корня), содержимое `public/` — в `~/public_html/test`, пути в `index.php` поправлены на `../../numeros-es/`.
2. **Проверить закрытость.** `curl https://домен/test/.env` → обязан вернуть 403/404. Если вернул содержимое — стоп, всё переделываем.
3. **Тестовая папка не индексируется.** `.htaccess`: Basic Auth + `Header set X-Robots-Tag "noindex, nofollow"` + `robots.txt`. Иначе Google съест тест и убьёт SEO будущего прода дублями.
4. **Очереди.** Supervisor нет. `QUEUE_CONNECTION=database` + cron ежеминутно:
   `* * * * * cd /home4/clsthmmy/numeros-es && /usr/local/bin/php artisan schedule:run >/dev/null 2>&1`
5. **Кэш.** Redis нет. `CACHE_STORE=file`, `SESSION_DRIVER=database`.
6. **Почта.** Не `mail()` — уйдёт в спам. Внешний SMTP + SPF/DKIM/DMARC.
7. **Бэкапы.** Свой cron `mysqldump` + ротация 14 дней + копия наружу. Автобэкапы Bluehost резервной копией не считаются.
8. **cPanel Terminal рвёт сессию каждые несколько минут.** Проверено дважды. Длинные команды (`composer install`) запускать через `nohup ... &` и смотреть лог, иначе процесс умрёт на полпути.
